===========================================================================
1. CONTEXT — What this app does
===========================================================================

This is a Flask-based Bengali/English audio transcription web app
("BM Transcriber"). It has:

- A basic transcription flow: audio -> OpenRouter (google/gemini-3.1-flash-lite)
  -> plain text. This route already works and must NOT be touched.
- A "sync" flow: audio -> transcription with word-level timestamps -> a
  browser player that highlights each word as it's spoken, using
  requestAnimationFrame synced to audio.currentTime.
- An auth system (Flask-Login + bcrypt + IP locking) — do not touch.
- An admin panel — do not touch.

THE PROBLEM: the sync flow currently estimates word timings mathematically
in timestamp_estimator.py (roughly: audio duration / word count), instead
of using real timestamp data. This produces drifting, inaccurate
highlighting — words fall out of sync with the audio, especially with
pauses or variable speech speed.

===========================================================================
2. GOAL — Hybrid transcription pipeline
===========================================================================

Replace math-based estimation with a two-model hybrid pipeline:

- Gemini (google/gemini-3.1-flash-lite via OpenRouter) -> produces the
  actual transcription TEXT. Keep this exactly as-is — its transcription
  quality is good and must not change.

- Whisper (openai/whisper-large-v3-turbo, ALSO via OpenRouter — same
  OPENROUTER_API_KEY you already use, no new provider account needed) ->
  used ONLY to extract accurate word-level TIMESTAMPS. Its own transcribed
  text is discarded/ignored; we only want its words: [{word, start, end}]
  array.
    - IMPORTANT: do NOT use openai/whisper-1 — that model on OpenRouter
      only returns plain text, no timestamps. You must use
      openai/whisper-large-v3-turbo, which OpenRouter's own model page
      states supports "timestamp granularities at word and segment levels."
    - This is the budget option — Turbo is faster and cheaper than the
      regular v3, yet still gives word-level timestamps.
    - Whisper's own transcription text must be thrown away entirely. We
      only keep its words array (the timing data). The final transcription
      text the user sees always comes from Gemini, never from Whisper.

- word_aligner.py -> aligns Gemini's words onto Whisper's timestamps using
  the existing Needleman-Wunsch-style alignment logic, so the final output
  has Gemini's correct text with Whisper's correct timing.

This must work exactly the same way for the existing "edit transcription"
flow — i.e. align_edited_words() should remain generic enough to be called
in both places:

1. Initial alignment: align_edited_words(whisper_words, gemini_text)
2. Post-edit realignment: align_edited_words(original_words, edited_text)

If the current function signature doesn't already support this dual use,
generalize it — don't fork into two separate functions unless there's a
strong reason to.

===========================================================================
3. TARGET ARCHITECTURE (reference — implement exactly this flow)
===========================================================================

User/Browser -> Flask (app.py): POST /transcribe-with-words (audio file)

Stage 1: Flask -> OpenRouter (/api/v1/chat/completions, Gemini model,
         base64 audio + Bengali prompt)
         Gemini -> Flask: transcription text

Stage 2: Flask -> OpenRouter (/api/v1/audio/transcriptions, Whisper Turbo
         model, base64 audio)
         Whisper Turbo -> Flask: { text, words: [{word, start, end}, ...] }
         (Whisper's own `text` field is discarded — only `words` is used)

Stage 3: Flask -> word_aligner.py: align_edited_words(whisper_words, gemini_text)
         word_aligner.py -> Flask: [{word, start, end, index, unchanged}, ...]

Flask stores result in SESSION_STORE & AUDIO_STORE

Flask -> Browser: { id, words, text, audio_url }

Browser: renderWords() -> DOM spans; rAF sync loop starts on play

--- Later, on edit ---
Browser -> Flask: PUT /api/session-transcriptions/<id> { edited_text }
Flask -> word_aligner.py: align_edited_words(original_words, edited_text)
word_aligner.py -> Flask: realigned words
Flask -> Browser: { words, text }
Browser: renderWords() -> re-render DOM
